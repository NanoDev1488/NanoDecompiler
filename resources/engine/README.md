# NanoDecompiler v1.1

Полноценный декомпилятор байткода Java для Bukkit/Spigot-плагинов. Не просто
дизассемблер — парсит `.class`-файлы вручную, строит граф потока управления,
символически исполняет байткод и **структурирует** его обратно в читаемый
Java (if/else, while/do-while/for, switch, try/catch — без единого `goto`).

Чистый Python 3, stdlib. Работает одинаково на Windows и в Termux (Android).

## Что это и чего НЕ делает

**Делает:**
- Парсит constant pool, поля, методы, атрибут `Code`, `LocalVariableTable`,
  аннотации (`RuntimeVisible/InvisibleAnnotations` + параметры),
  `BootstrapMethods` (лямбды/method references), `InnerClasses`.
- Строит CFG (граф потока управления) из инструкций байткода + таблицы
  исключений.
- Символически исполняет каждый метод как стек-машину, восстанавливая
  выражения: арифметика, вызовы методов, `new`, касты, конкатенация строк,
  инкременты, тернарники, лямбды и method references.
- **Структурирует** управляющий поток через дерево доминаторов/
  постдоминаторов в `if`/`else`, `while`/`do-while`/`for`, `switch`
  (включая `switch(String)` через паттерн `hashCode`+`equals`), `try`/`catch`.
- Деобфусцирует имена классов/методов/полей/пакетов (мусорные короткие
  имена → читаемые: `ClassA1`, `method3`, `field7`, `pkg2`...) —
  консервативно, не трогает то, что и так похоже на осмысленное имя.
- Определяет и вычленяет известные шейднутые библиотеки (JDBC-драйверы,
  JSON/YAML, логирование, HTTP-клиенты и т.д., включая релоцированные через
  maven-shade `<relocations>`) — не декомпилирует их, а прописывает как
  обычную maven-зависимость в сгенерированном `pom.xml`.
- Генерирует `pom.xml`, `MAPPING_RU.txt` (было → стало) и `README_RU.txt`
  с отчётом и статистикой точности прямо в папке результата.

**Принципиально НЕ делает:**
- **Никогда не печатает код, в котором не уверен на 100%.** Если для
  конкретного метода встретился непривычный/редкий паттерн байткода —
  декомпилятор откатывается на честный дизассемблированный листинг именно
  для этого метода (с пояснением-комментарием), а не гадает и не выдаёт
  код, который может быть неверным. Остальные методы это не затрагивает.
  Это ключевой архитектурный принцип: лучше «не смог», чем «соврал».
- Не декомпилирует чужой закрытый код общего назначения (например, код
  самого Minecraft/Mojang) — только сторонние плагины поверх публичного
  Bukkit/Spigot API.

## Быстрый старт

**Linux / macOS / Termux:**
```bash
python3 main.py plugin.jar [output_dir]
```
Если `output_dir` не указан — создаётся `<имя_jar>_decompiled` рядом.

**Windows:**
Просто запусти `run.bat` (двойной клик) — откроется GUI. Можно перетащить
`.jar`-файл прямо на `run.bat` — GUI откроется с уже подставленным путём и
сразу начнёт декомпиляцию.

## Режимы запуска

### CLI (Linux/macOS/Termux)

```bash
python3 main.py plugin.jar [output_dir]
```
Печатает цветной прогресс, баннер, проверку окружения (java/maven) и
итоговую статистику. Результат — папка с исходниками `.java`, `pom.xml`,
`MAPPING_RU.txt`, `README_RU.txt`.

### GUI (Windows)

Три переключаемые темы оформления, **по умолчанию — MD3** (нужен
`pip install flet`; если flet не установлен при первом запуске — тихо
откатится на «Неоновую», затем на «Сырую», см. ниже):

| Тема   | Движок                          | Зависимости                  |
|--------|----------------------------------|-------------------------------|
| MD3 (по умолчанию) | Flet / Material Design 3 (`gui_md3.py`) | `pip install flet`   |
| Неоновая | customtkinter (`gui_neon.py`) | `pip install customtkinter pillow` |
| Сырая  | classic ttk (`gui_raw.py`)      | нет (только stdlib)           |

Тема переключается на экране «Настройки» внутри самого GUI. Если для
выбранной темы не хватает зависимостей — приложение откатывается по цепочке
`md3 → neon → raw` (первая, для которой всё на месте) и явно показывает
причину (и в консоли, и всплывающим окном — см.
[Диагностика проблем](#диагностика-проблем)). Значит, если после установки
`flet` не забыт — при самом первом запуске (когда `settings.json` ещё нет)
пользователь сразу увидит MD3, а не старую tkinter-тему.

Настройки (тема + папка результатов по умолчанию) хранятся в
`%LOCALAPPDATA%\NanoDecompiler\settings.json`.

`gui.py` — единственная точка входа/диспетчер; `gui_raw.py`/`gui_neon.py`/
`gui_md3.py` никогда не запускаются напрямую пользователем.

### API-режим (для интеграции с другими приложениями)

Два варианта — можно использовать любой или оба сразу.

**Разовый вызов, JSON в stdout:**
```bash
python3 main.py plugin.jar [out_dir] --json-output
```
Печатает **только** JSON (без баннера/цвета/прогресс-бара), код возврата
`0` (успех) или `1` (ошибка). Пример ответа:
```json
{
  "status": "ok",
  "out_dir": "plugin_decompiled",
  "elapsed_sec": 0.37,
  "stats": {
    "classes_total": 167,
    "classes_parsed": 167,
    "decompiled_methods": 296,
    "fallback_methods": 22,
    "decompiled_pct": 93.08,
    "fallback_reasons": { "...": 11 },
    "import_conflicts": { "Entry": ["a.b.Entry", "c.d.Entry"] },
    "library_names_hit": ["org.xerial:sqlite-jdbc"]
  }
}
```

**HTTP-сервер (stdlib `http.server`, без сторонних зависимостей):**
```bash
python3 main.py --api-server [--host 127.0.0.1] [--port 8791]
```
- `POST /decompile` — тело запроса: JSON `{"jar_path": "...", "out_dir": "..."}`
  ИЛИ сырые байты `.jar` напрямую (`Content-Type: application/java-archive`
  или `application/octet-stream`, опционально `?out_dir=...` в query-строке).
- `GET /health` — проверка живости.

**Гарантия:** в API-режиме GUI никогда не импортируется и не открывается —
даже на Windows, даже без аргументов. Проверяется первой строкой в
`main()`, до ветки `platform.system() == "Windows"`.

⚠️ Сервер слушает `127.0.0.1` по умолчанию — аутентификации/токена нет,
это рассчитано на локальную интеграцию. Если когда-нибудь понадобится
слушать не только localhost — обязательно добавить токен в заголовок.

## Установка Java/Maven по требованию

Проверка окружения (`java`/`mvn`) происходит перед декомпиляцией
автоматически и ничего не ставит сама — только сообщает, чего не хватает.

Если чего-то нет — можно поставить одной командой, **portable, без прав
администратора**, в локальную папку рядом с настройками (никак не трогает
системный `PATH`):

```bash
python3 main.py --install-tools            # поставит и JDK, и Maven, если не хватает
python3 main.py --install-tools=jdk        # только JDK
python3 main.py --install-tools=maven      # только Maven
```

- JDK — Eclipse Temurin (Adoptium API), LTS-версия 17.
- Maven — последняя стабильная версия с `dlcdn.apache.org`.
- Устанавливается в `%LOCALAPPDATA%\NanoDecompiler\tools` (Windows) или
  `~/.nanodecompiler/tools` (Linux/Termux).
- Проверка окружения потом видит эту папку сама, даже если `java`/`mvn`
  так и не попали в системный `PATH`.

На Windows проверка также умеет находить java/maven, установленные, но ещё
не подхватившиеся текущим процессом (классическая ситуация «поставил — а
PATH обновится только после перезагрузки») — читает актуальный `PATH` прямо
из реестра и проверяет типичные пути установки (`JAVA_HOME`, `MAVEN_HOME`,
`Program Files\...`, scoop).

## Что на выходе

В указанной папке результата:
- `src/...` (или аналогичная структура пакетов) — восстановленные `.java`.
- `pom.xml` — сгенерированный/восстановленный, с зависимостями на известные
  шейднутые библиотеки.
- `MAPPING_RU.txt` — таблица «было → стало» для всех переименованных
  классов/методов/полей/пакетов.
- `README_RU.txt` — отчёт: сколько методов восстановлено полностью, сколько
  ушло в честный fallback на байткод и почему, коллизии импортов, найденные
  библиотеки, общая оценка «крутизны» декомпиляции.

## Как это работает (архитектура)

Плоская структура — все `.py`-файлы в одной папке, без пакетов:

```
main.py         оркестрация (process_jar/process_jar_with_stats), CLI/GUI
                entry point, консольный вывод, проверка java/maven,
                детект известных библиотек и релокаций
classfile.py    парсер .class (constant pool, поля, методы, Code,
                LocalVariableTable, аннотации, BootstrapMethods, InnerClasses)
ir.py           декодирует сырой байткод в список инструкций
cfg.py          строит CFG из инструкций + таблицы исключений
stackvm.py      символическое исполнение блока: байткод -> дерево выражений
engine.py       decompile_method_body() - весь пайплайн одного метода,
                все проверки DecompileAbort
structure.py    CFG -> структурированное дерево (if/while/for/switch/try)
emit.py         AST -> текст Java
ast_nodes.py    узлы AST
javatypes.py    дескрипторы JVM -> Java-типы, эвристика "похоже на обфускацию"
pom_builder.py  генерация pom.xml, таблица известных библиотек
verify.py       статистика точности, баланс скобок, коллизии импортов
switchmap.py, disassembler.py, opcodes.py   вспомогательные модули
toolinstaller.py   portable-закачка JDK/Maven по требованию
api.py          JSON CLI-режим и HTTP API-сервер
gui.py          диспетчер GUI-тем (запускает нужный движок)
gui_raw.py      GUI-тема "Сырая" (classic ttk)
gui_neon.py     GUI-тема "Неоновая" (customtkinter)
gui_md3.py      GUI-тема "MD3" (Flet, Material Design 3)
gui_common.py   общая логика для всех трёх GUI-тем (настройки, сводка по
                jar, захват консольного вывода) - не тянет тяжёлые тулкиты
run.bat/run.sh  точки входа
icon.ico/icon.png   иконка (сгенерирована через PIL, не внешний файл)
```

**Ключевой принцип, который нельзя нарушать:** если декомпилятор не уверен
на 100% в конкретном методе — он обязан откатиться на честный
дизассемблированный листинг байткода (`DecompileAbort` в `engine.py`)
вместо того, чтобы напечатать код, который может быть неправильным.

## Тестирование / регрессия

В `test_jars/` лежат три тестовых jar'а разной сложности:
- `EryBuyer-v1.jar` — шейдит sqlite-jdbc, интерфейсы, enum, synchronized.
- `DeathUtils-1_0.jar` — шейдит sqlite-jdbc **релоцированный** (проверяет
  детект релокаций).
- `GlowClans-0_1-fork.jar` — самый ценный тестовый jar, реальный лог
  `mvn clean package` нашёл 6+ реальных багов движка. Есть `@NotNull`
  (JetBrains).

Прогон регрессии:
```bash
for j in test_jars/*.jar; do
    python3 main.py "$j" "/tmp/out_$(basename "$j" .jar)"
done
```
Базовые цифры («Полностью восстановлены», актуально на момент этой сборки):
**EryBuyer ~93.1%, DeathUtils ~93.8%, GlowClans ~96.8%.** Заметная просадка
на нескольких jar'ах после правки — сигнал, что новый фикс/проверка ложно
срабатывает (см. `HANDOFF_2_FIXES_HISTORY.md`, там подробная история двух
таких случаев).

## Сборка в .exe (Windows)

Если нужна **только** тема «Сырая» (без customtkinter/flet):
```
pyinstaller --onefile --windowed --icon=icon.ico --name NanoDecompiler main.py
```

Если нужны **все три темы** в одном .exe — зависимости нужно явно указать,
иначе PyInstaller их не подхватит (динамические `import gui_neon`/
`import gui_md3` не всегда видны статическому анализатору):
```
pip install customtkinter pillow flet
pyinstaller --onefile --windowed --icon=icon.ico --name NanoDecompiler ^
    --hidden-import=customtkinter --hidden-import=PIL --collect-all flet ^
    main.py
```

PyInstaller не кросс-компилирует — собирать нужно реально на Windows (если
под рукой только Termux/Linux — вариант: GitHub Actions с
`windows-latest`-раннером, пуш из Termux, скачать `.exe` из Artifacts).

## Известные ограничения

- **Дженерики**: полный парсинг атрибута `Signature` не реализован —
  объявления полей/переменных с generic-параметрами печатаются как raw-тип
  (`Map` вместо `Map<String, UUID>`). Касты на границах вызовов это
  компенсируют, компилируется корректно, просто не так красиво.
- **`synchronized(){}`**: настоящая свёртка `monitorenter`/`monitorexit` не
  реализована — если найден несвёрнутый монитор, честно откатывается на
  байткод вместо потери семантики блокировки.
- **`try`/`finally`**: восстанавливается как несколько `catch(Throwable)` с
  дублированием кода вместо одного `finally {}` — семантически верно,
  компилируется, но не идиоматично для чтения.
- **GUI-темы «Неоновая»/«MD3»**: визуально не проверялись на реальном
  дисплее (только синтаксически) — см. раздел «Диагностика проблем» ниже,
  если тема не переключается или выглядит не так.
- Известный баг движка (не регрессия этой сборки — воспроизводится и на
  базовых цифрах): в `GlowClans` 2 метода откатываются с
  `AttributeError: 'SwitchStmt' object has no attribute 'expr'`.

## Диагностика проблем

**«Не могу изменить тему в GUI» / выбрал неоновую или MD3, а осталась
сырая:**
Скорее всего собранный `.exe` не включает `customtkinter`/`flet` (см.
раздел «Сборка в .exe» — нужны `--hidden-import`/`--collect-all`, простой
`pip install` уже собранному `.exe` не помогает). Приложение теперь
показывает всплывающее окно с точной причиной отката при каждом таком
случае — обрати внимание на текст сообщения. Если после пересборки с
правильными флагами всё ещё не переключается — временно убери `--windowed`
из команды сборки, чтобы увидеть полную консоль, и пришли лог.

**«На Windows всегда пишет, что Maven/Java отсутствует», хотя точно
установлены:**
Частая причина — `PATH` обновился в реестре установщиком, но текущий
процесс (особенно если приложение запущено двойным кликом через
explorer.exe) унаследовал старое окружение и не видит изменения до
перезахода в систему/перезагрузки. Проверка теперь дополнительно читает
свежий `PATH` из реестра и типичные пути установки — если находит именно
так, явно пишет «найден(а) вне PATH» + путь + подсказку про перезагрузку.
Если и это не помогает — попробуй `python3 main.py --install-tools`,
это ставит portable-копию рядом, независимо от системного `PATH`.

**Любая другая непонятная просадка после правки движка** — сначала смотри
`stats.fallback_reasons` в `README_RU.txt`/JSON-выводе (какая именно
причина отката выросла), не гадай и не откатывай фикс вслепую (см. историю
в `HANDOFF_2_FIXES_HISTORY.md` — там разбор именно такого случая).

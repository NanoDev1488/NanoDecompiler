## 2. ГОТОВО: Режим API (для интеграции с чужим приложением)

Сделаны ОБА варианта, как и предполагалось в исходной задаче (см. ниже):
- `python3 main.py plugin.jar [out_dir] --json-output` - разовый вызов,
  печатает ТОЛЬКО JSON в stdout (без баннера/цвета/прогресс-бара), код
  возврата 0/1.
- `python3 main.py --api-server [--host H] [--port 8791]` - локальный
  HTTP-сервер на `http.server` (stdlib, без зависимостей): `POST
  /decompile` (тело JSON `{"jar_path", "out_dir"}` ИЛИ сырые байты .jar с
  `Content-Type: application/java-archive`/`application/octet-stream` +
  опциональный `?out_dir=`), `GET /health`.

Новый файл `api.py`. Ключевая гарантия из требования пользователя: оба пути
проверяются в `main._try_handle_api_mode()` - вызывается ПЕРВОЙ строкой в
`main()`, ДО ветки `platform.system() == "Windows"`, так что `gui.py`
физически не импортируется в этом режиме - проверено юнит-тестом (spy на
`__import__`) и симуляцией полного вызова `main()` с замоканным
`platform.system() -> "Windows"`.

`process_jar()` (main.py) разбит на `process_jar()` (совместимость,
возвращает как раньше только `out_dir`) + `process_jar_with_stats()`
(возвращает `(out_dir, ProjectStats)`) - используется как GUI/CLI, так и
`api.py` без дублирования пайплайна. `api.stats_to_dict()` сериализует
`ProjectStats` в JSON-совместимый dict (в частности `library_names_hit` -
это `set`, приведено к отсортированному списку; ключ `None` в
`fallback_reasons` заменён на строку `"unknown"`).

Проверено на всех трёх test_jars (оба варианта - JSON-путь и raw-bytes
через HTTP) - статистика в JSON совпадает с README_RU.txt (93.8/93.1/96.8%).
Полная регрессия (см. HANDOFF_1) после этих изменений - без просадок.

НЕ СДЕЛАНО (можно доделать по запросу): аутентификация/токен на HTTP-сервер
(сейчас любой локальный процесс может дёрнуть `/decompile` - для
localhost-интеграции с "тем самым приложением друга" это, скорее всего,
приемлемо, но если сервер когда-нибудь будет слушать не только 127.0.0.1 -
обязательно добавить хотя бы токен в заголовке).

## 3. ГОТОВО (CLI-часть): Локальная автозакачка JDK/Maven "по требованию"

Сделан вариант Б - НИЧЕГО не ставится молча, только по явному запросу. Новый
файл `toolinstaller.py` (без зависимостей сверх stdlib - urllib/zipfile/
tarfile):
- `install_jdk()` - Adoptium API v3, `/v3/binary/latest/17/ga/{os}/{arch}/
  jdk/hotspot/normal/eclipse` (редирект прямо на архив - zip на Windows,
  tar.gz на linux/mac, включая Termux).
- `install_maven()` - номер версии из `maven-metadata.xml` на repo1.maven.org
  (машиночитаемо, не парсинг HTML dlcdn), архив (`-bin.zip`) качается с
  `dlcdn.apache.org/maven/maven-3/...`, есть запасной номер (3.9.9), если
  dlcdn ещё не успел раздать версию из свежего metadata.xml.
- Ставится portable, БЕЗ прав администратора, в `<app_data_dir>/tools`
  (тот же каталог, что и `gui_common.get_app_data_dir()`). `find_local_java()`/
  `find_local_maven()` сканируют папку по префиксу.
- `check_java_maven()` (main.py) проверяет ОБА источника - PATH, потом
  локальную папку - и печатает, откуда именно нашлась java/mvn.
- Новый режим: `python3 main.py --install-tools` (JDK+Maven) или
  `--install-tools=jdk`/`--install-tools=maven` (только одно). Прогресс
  печатается построчно. Ошибка установки одного инструмента не мешает
  попытке поставить другой (`install_missing()` собирает ошибки в список).

ПРОВЕРЕНО: полная регрессия без просадок (93.75/93.08/96.76%). Логика
скачивания+распаковки+поиска покрыта функциональным тестом с локальным
HTTP-сервером (синтетические zip/tar.gz вместо реальных JDK/Maven) - оба
формата архива, плюс проверка что бинарник после установки физически
исполняем (`chmod +x` на не-Windows). В песочнице реальной сети нет
(adoptium.net/dlcdn.apache.org возвращают 403 через прокси) - живой факт
"настоящий JDK/Maven скачался и работает" НЕ проверен, обязательно
прогнать `--install-tools` на реальной машине (Windows/Termux) перед
релизом.

## СБОРКА В .exe (для пользователя, Windows)

ВАЖНО (см. раздел "ИСПРАВЛЕННЫЕ БАГИ ЭТОЙ СЕССИИ" выше про тему "не
переключается") - если нужны ВСЕ три темы в одном .exe, команда сборки
ДОЛЖНА явно перечислить опциональные зависимости, иначе PyInstaller не
включит их (динамические `import gui_neon`/`import gui_md3` внутри
условной ветки часто не подхватываются статическим анализом):

```
pip install customtkinter pillow flet
pyinstaller --onefile --windowed --icon=icon.ico --name NanoDecompiler ^
    --hidden-import=customtkinter --hidden-import=PIL --collect-all flet ^
    main.py
```

Если нужна ТОЛЬКО "сырая" тема (без customtkinter/flet вообще) - исходная
команда без доп. флагов по-прежнему подходит:
`pyinstaller --onefile --windowed --icon=icon.ico --name NanoDecompiler main.py`

PyInstaller НЕ кросс-компилирует - собирать нужно РЕАЛЬНО на Windows (у
пользователя нет Windows-машины под рукой - предлагали GitHub Actions
workflow с `windows-latest` runner для сборки в облаке, пуш из Termux,
скачать готовый .exe из Artifacts).

## ИСПРАВЛЕННЫЕ БАГИ ЭТОЙ СЕССИИ (по репорту пользователя с реальной Windows-машины)

**1. "Не могу изменить тему"** - НАЙДЕНА вероятная причина: сборка .exe
через `--windowed` означает, что консоли НЕТ ВООБЩЕ - когда `gui.py`
откатывался с md3/neon на raw из-за отсутствующей зависимости
(`_resolve_theme`), диагностическое сообщение печаталось через `print()`,
которое просто улетало в никуда - пользователь видел только "выбрал
неоновую/MD3, сохранил, а получил снова сырую", без единой подсказки
почему. Похоже, что именно так и произошло: `customtkinter`/`flet` скорее
всего не были явно включены в PyInstaller-сборку (см. предыдущую сессию -
"НЕ ПРОВЕРЕНО ВИЗУАЛЬНО" в п.1 выше), а `pip install` ПОСЛЕ того, как .exe
уже собран, ничего не даёт - .exe самодостаточен, в него зашито только то,
что было на месте В МОМЕНТ СБОРКИ.

Исправлено: `gui.py::_resolve_theme` теперь ДОПОЛНИТЕЛЬНО показывает
`messagebox.showwarning` (голый tkinter, работает без консоли и без
customtkinter/flet) с точной причиной отката и, если это была MD3 -
явным намёком на `--hidden-import`/`--collect-all` при пересборке. Плюс
исправлена сама команда сборки (см. выше) - если ей не следовали раньше,
это, вероятно, и есть первопричина.

Если после пересборки с `--hidden-import`/`--collect-all` тема ВСЁ РАВНО
не переключается - значит первопричина другая, тогда стоит собрать полный
stdout (например, временно убрать `--windowed` из сборки, чтобы увидеть
консоль) и прислать для дальнейшей диагностики.

**2. "На Windows всегда пишет что maven отсутствует"** - НАЙДЕНА вероятная
причина: классический для Windows кейс - `shutil.which()` смотрит только
`PATH` ТЕКУЩЕГО процесса, а `PATH` обновляется в реестре установщиком
Maven, но уже запущенные/унаследованные окружения (в т.ч. explorer.exe -
родитель для двойного клика по .exe) остаются со старым `PATH` до
перезахода/перезагрузки. Плюс отдельная частая причина - Maven на Windows
нередко "устанавливают" простой распаковкой zip куда попало, вообще не
трогая `PATH`.

Исправлено (`toolinstaller.py::resolve_tool_path`, используется в
`main.py::check_java_maven`): если голый `shutil.which()` не находит -
на Windows дополнительно (а) читает АКТУАЛЬНЫЙ `PATH` прямо из реестра
(`HKCU\Environment` и `HKLM\...\Session Manager\Environment`, не полагаясь
на кэш процесса) и (б) проверяет типичные места установки
(`JAVA_HOME`/`MAVEN_HOME`/`M2_HOME`, `Program Files\Apache\maven*`,
`Program Files\Java\...`, scoop). Если найдено ТОЛЬКО через этот
запасной путь - в выводе явно написано "найдена вне PATH" + сама причина
(что PATH в процессе устарел) + путь, который сработал.

ПРОВЕРЕНО: логика `_well_known_dirs`/`resolve_tool_path` протестирована
функционально (переменные окружения `JAVA_HOME`/`MAVEN_HOME`, симуляция
`platform.system() -> "Windows"` на linux-песочнице) - находит бинарник
по переменной окружения корректно. Чтение из реестра (`winreg`) физически

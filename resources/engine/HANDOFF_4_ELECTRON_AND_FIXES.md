# HANDOFF_4: Electron-клиент, разделение на 2 продукта, CI/CD, баги с реального теста

Продолжение HANDOFF_1/2/3 (те - про сам движок-декомпилятор, ещё до
Electron; актуальны по-прежнему, архитектурные принципы не менялись).
Этот файл - про всё, что случилось ПОСЛЕ: переход с трёх Python GUI-тем
(tkinter/customtkinter/flet) на Electron+React, разделение на клиент/API,
CI/CD с автопубликацией в GitHub Releases, и баги, найденные на реальном
тесте пользователя (не в песочнице).

## Структура проекта СЕЙЧАС (один репозиторий)

```
repo-root/
  package.json, electron/, src/, index.html, tsconfig.json, vite.config.ts
                                              <- Electron+React клиент
  resources/engine/*.py                       <- движок (тот же, что в HANDOFF_1-3)
  test_jars/EryBuyer-v1.jar                   <- для CI-проверки (не для конечного пользователя)
  .github/workflows/build-and-release.yml     <- собирает и клиент, и API, публикует ОДИН релиз
  .gitignore
```

Пользователь уже смёржил свой старый (движок-только) репозиторий в эту
структуру - см. `GIT_MERGE_INSTRUCTIONS.md` в корне, если понадобится
объяснить кому-то ещё раз.

## Два продукта (НЕ путать при работе)

1. **Клиент** (`NanoDecompiler-Client-Setup.exe`, tag `v1.2`) - Electron+React,
   полноценный GUI. `electron/main.ts` спавнит `resources/engine/main.py`
   с флагом `--headless` (ОБЯЗАТЕЛЕН на Windows - без него движок пытается
   открыть старый GUI, которого больше физически нет - см. "Убрали Python
   GUI" ниже).
2. **API** (`NanoDecompilerAPI.exe`, tag `v1.0`, entry point `api_cli.py`) -
   ГОЛЫЙ API для встраивания в чужой проект ("4 в 1" декомпилер пользователя,
   тоже Electron). НЕ сервер - не висит в фоне. Вызов:
   `NanoDecompilerAPI.exe plugin.jar [out_dir]` -> одна строка JSON в stdout.
   `api_cli.py` - ОТДЕЛЬНЫЙ от `main.py` файл специально: не импортирует
   ничего GUI-связанного даже транзитивно, физически не может показать окно.
3. (Бонус) `NanoDecompilerCLI.exe` - тот же движок (`main.py`), что и в
   клиенте, но как отдельный CLI-инструмент со всеми флагами
   (`--api`/`--json-output`/`--api-server`/`--install-tools`/
   `--install-tools-json`/`--jar-summary`). Публикуется в том же релизе.

Все три собираются и публикуются АВТОМАТИЧЕСКИ на каждый `push` в `main`
через `build-and-release.yml` - три job'а (`build-client`, `build-api`,
`publish-release`), последний ждёт первые два и публикует ОДИН GitHub
Release с текстовым описанием + всеми файлами. Тег статический (`v1.2` -
см. `env.RELEASE_TAG` в начале workflow) - повторный push с той же версией
обновляет файлы в существующем релизе, а не плодит дубликаты.

## Убрали Python GUI (gui.py/gui_raw.py/gui_neon.py/gui_md3.py)

Эти файлы физически удалены из `resources/engine/` в этой сессии - Electron
теперь единственный GUI, старые три tkinter/customtkinter/flet темы были
мёртвым весом. `gui_common.py` ОСТАВЛЕН - он не про GUI-тулкиты, а про общую
логику (`jar_summary()`, `open_in_vscode()`, работу с настройками), и
сейчас АКТИВНО используется через `main.py --jar-summary` (см. ниже).

Важное следствие: `main.py::main()` раньше на Windows без аргументов
пытался `from gui import run_gui` - с удалением gui.py это упало бы
`ImportError`. Заменено на понятное usage-сообщение, отправляющее к
Electron-клиенту (см. `main.py`, ветка `platform.system() == "Windows" and
"--headless" not in sys.argv[1:] and len(sys.argv) < 2`).

## Баги, найденные на реальном Windows-тесте (не в песочнице) - ВСЕ ПОЧИНЕНЫ

1. **`npm ci` падала везде** - в репозитории не было `package-lock.json`
   (собирался `package.json` вручную, без сети). Поменяли на `npm install`
   в workflow. На будущее: сгенерировать нормальный lock-файл и вернуть
   `npm ci` (быстрее, детерминированнее).

2. **`UnicodeEncodeError` в PyInstaller-сборках на GitHub Actions** -
   Windows-раннер работает в английской локали (`cp1252`), не умеющей
   кириллицу, а наш JSON/консольный вывод её содержит. Починено
   принудительным `sys.stdout.reconfigure(encoding="utf-8",
   errors="replace")` в начале `main.py` И `api.py` (оба места нужны -
   разные entry points). Проверено репродукцией с эмуляцией cp1252-потока.

3. **`icon.ico must be at least 256x256`** (electron-builder/NSIS) - старый
   `icon.ico` был только 16x16. Пересобран из `icon.png` (512x512) как
   многослойный (16/24/32/48/64/128/256).

4. **electron-builder сам пытался публиковать релиз** (`GitHub Personal
   Access Token is not set`) - CI-детект внутри electron-builder триггерил
   попытку автопаблиша, а у нас релиз публикует ОТДЕЛЬНЫЙ job. Починено:
   `"publish": null` в `package.json` build-конфиге + `--publish=never` в
   npm-скрипте `dist` (двойная защита).

5. **PowerShell-поиск собранного инсталлятора рисковал схватить
   `win-unpacked/NanoDecompiler.exe`** (сырой распакованный exe) вместо
   самого инсталлятора. Починено - сперва ищет по имени с "Setup", запасной
   вариант явно исключает `win-unpacked/`.

## Баги, найденные на реальном ТЕСТЕ ДРУГА (2-3 плагина, не в песочнице)

1. **24 МБ лишнего в результате** - `native/` папка от sqlite-jdbc (бинарники
   под 9 платформ) не была распознана как известная библиотека, потому что
   библиотека была relocated (`org.sqlite` -> `net.mlsac.libs.sqlite`), а
   детект релокации полагался ТОЛЬКО на `<relocations>` из pom.xml САМОГО
   ПЛАГИНА - а такого pom.xml в этом конкретном jar'е не было забандлено
   вообще (частый реальный кейс, не редкость). Заодно раздекомпилились
   RxJava3 и часть Guava (`com.google.thirdparty.publicsuffix`,
   `com.google.errorprone.annotations`) - их просто не было в `KNOWN_LIBS`.

   Починено:
   - Новый fallback-детект `_signature_relocated_prefixes()` в `main.py` -
     ищет характерные ПУТИ РЕСУРСОВ (для sqlite-jdbc: `.../native/{OS}/{arch}/
     (lib)?sqlitejdbc.{so,dll,dylib}`) независимо от pom.xml. Расширяемо -
     можно добавлять сигнатуры для других часто shaded библиотек по мере
     обнаружения (структура `_SIGNATURE_PATTERNS` в `main.py`).
   - `KNOWN_LIBS` (`pom_builder.py`) пополнен: `io.reactivex.rxjava3`,
     `com.google.thirdparty` (guava), `com.google.errorprone.annotations`.
   - Проверено на ТОЧНЫХ путях из реального jar'а друга - детект срабатывает,
     код самого плагина не задевает. Регрессия на трёх штатных test_jars -
     без изменений (93.75/93.08/96.76%).

   ⚠️ Это НЕ исчерпывающее решение - список известных библиотек и их
   сигнатур конечен по определению, каждый новый плагин может шейдить
   что-то ещё не встречавшееся. Если пользователь снова пришлёт раздутый
   результат - первым делом смотреть, какие пакеты декомпилировались
   ПОЛНОСТЬЮ (много классов, похожих на стороннюю библиотеку по
   именам/структуре) и добавлять в `KNOWN_LIBS`/`_SIGNATURE_PATTERNS`.

2. **"Открыть в VS Code" кнопки не было в Electron-клиенте** - хотя
   аналогичная функция (`open_in_vscode()`) существовала в старом
   `gui_common.py`. Добавлена IPC-ручка `shell:openInVSCode` в
   `electron/main.ts` (`spawn("code", [path], {shell: true})`) + кнопка в
   `App.tsx`, показывается рядом с "Открыть папку" после успешной
   декомпиляции.

3. **Карточка сводки по jar пропала** (была задумана по референсу - "4-й
   скрин", фиолетовая карточка с CFR/Procyon/FernFlower - референс был
   только по СТИЛЮ карточки, не по выбору декомпилятора, у нас его нет и
   не будет, движок всегда один свой). Данные для карточки уже считались в
   `gui_common.jar_summary()` (имя, размер, Java-версия, число классов,
   число пакетов, имя плагина из plugin.yml) - просто не были подключены к
   Electron. Добавлено:
   - `main.py --jar-summary plugin.jar` - печатает JSON.
   - IPC `jar:summary` в `electron/main.ts`.
   - Карточка `.jar-card` в `App.tsx` + стили в `styles.css`.

4. **Баг "Новая папка"** - диалог выбора папки результата открывался
   ПУСТЫМ, без подсказки имени. Если пользователь нажимал "Новая папка" в
   Проводнике не глядя и не переименовывал - оставалось буквально имя
   "Новая папка" (это баг UX, не движка - движок использует ЛЮБОЙ путь,
   который ему передали). Починено: `dialog:selectOutDir` в
   `electron/main.ts` теперь принимает `defaultPath` (вычисляется в
   `App.tsx::defaultOutPathFor()` из имени jar'а - `<jar>_decompiled` рядом
   с самим jar'ом), диалог открывается уже с подставленным именем.

5. **Кнопка "Установить java/maven?" добавлена** (не баг, доп. фича по
   запросу) - раньше диагностика "Не хватает: maven" была только текстом в
   логе с советом запустить команду в терминале, которого у пользователя
   приложения может не быть. Добавлено:
   - `main.py --install-tools-json[=jdk|maven]` - то же самое, что
     `--install-tools`, но NDJSON вместо текста с `\r`-прогрессом (для
     реального % в UI вместо парсинга текста).
   - IPC `tools:install` в `electron/main.ts`, стримит события `tools:progress`.
   - В `App.tsx`: строка лога "Не хватает: ..." парсится регуляркой
     (`parseMissingTools`), рядом рисуются мини-кнопки "Да/Нет" (MD3-чип,
     см. `.install-prompt`/`.btn-mini` в `styles.css`) - не отдельный попап,
     а прямо на месте, как просили.

## Не проверено / требует внимания

- **Все TS-изменения этой сессии проверены ТОЛЬКО офлайн-тайпчеком с
  самодельным shim'ом** (`@types/react`/`@types/node`/`electron` недоступны
  без сети в песочнице) - логически проверено, но `npm run build` +
  `electron-builder` вживую не прогонялись мной ни разу. Первая реальная
  проверка - следующий push пользователя и CI-лог.
- **Сигнатурный детект relocated-библиотек** пока покрывает только
  sqlite-jdbc (единственный реально столкнувшийся кейс) - расширять по
  мере поступления новых раздутых результатов от пользователя.
- **`package-lock.json` всё ещё отсутствует** - `npm install` в CI работает,
  но не детерминирован между запусками так же строго, как `npm ci` был бы.
- Старые пункты 4-6 из `HANDOFF_3_TODO.md` (Signature-дженерики,
  `synchronized(){}`, идиоматичный `try/finally`) - всё ещё не начаты,
  не приоритет.
- Известный баг движка (воспроизводится с самого начала, не регрессия):
  в `GlowClans` 2 метода откатываются с `AttributeError: 'SwitchStmt'
  object has no attribute 'expr'`.

## Ключевой принцип (не менялся, повторяю для новых сессий)

Если декомпилятор не уверен на 100% в конкретном методе - он ОБЯЗАН
откатиться на честный дизассемблированный листинг байткода вместо того,
чтобы напечатать код, который может быть неверным (`DecompileAbort` в
`engine.py`). Этот принцип важнее любого процента "успешно
декомпилировано" - не оптимизировать один в ущерб другому.

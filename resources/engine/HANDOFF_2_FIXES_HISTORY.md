# HANDOFF 2/3: ИСТОРИЯ ФИКСОВ - NanoDecompiler v1.1

Сжато, по категориям. Если нужны детали - смотри сам код в указанных файлах.

## Импорты/типы

- Коллизии simple-имён импортов (напр. `org.bukkit.ChatColor` vs
  `net.md_5.bungee.api.ChatColor` в одном файле) - двухпроходная схема через
  `mark_type()`/`resolve_type_markers()` (javatypes.py) - первый встреченный
  тип оставляет import+короткое имя, остальные - FQN без импорта.
- `Type[].class` печатался как невалидный `[Ljava/lang/reflect/Type;.class` -
  исправлено через `_array_type_str()`.
- Убрана лишняя самоквалификация: `this.field`/`this.method()` -> `field`/
  `method()` когда нет коллизии с локальной переменной (`emit.py::
  set_shadow_context`); `OwnClass.staticField` -> `staticField` для СВОИХ же
  static-полей/методов (`emit.py::set_current_class`) - это было РЕАЛЬНЫМ
  багом компиляции для blank final static полей (javac не признаёт
  квалифицированное присваивание "тем самым" присваиванием blank final).

## Локальные переменные

- LocalVariableTable теперь читается и используется (реальные имена
  переменных вместо `argN`/`varN`), если jar собран с отладочной информацией.
- Сигнатура метода строится ПОСЛЕ декомпиляции тела (не до), чтобы имена
  параметров совпадали с тем, что реально использует тело (было
  рассогласование: сигнатура `arg0`, тело уже `player`).
- Параметры не попадали в "уже объявлено" -> переприсваивание параметра
  печаталось как повторное объявление ("variable already defined").
- Catch-переменные (`e1`, `e2`...) - счётчик теперь ОБЩИЙ на весь метод
  (был per-try, коллизии при вложенных try/catch).
- **Утечка области видимости** (`_has_escaping_local_decl`/
  `_hoist_escaping_locals` в engine.py) - переменная, объявленная внутри
  if/while/for/try/switch, но используемая ПОСЛЕ этого блока (javac-идиома
  switch(String) через hashCode; guard-clause `if(cond){setup} else
  {return;} <используем setup>`) - теперь ПОДНИМАЕТСЯ (голое объявление
  наружу + `LocalDecl` внутри превращается в присваивание), а не просто
  ловится и откатывается.

  ВАЖНО, ЧТОБЫ НЕ НАСТУПИТЬ ПОВТОРНО: проверка escape - через
  `_collect_shallow_referenced_names` (НЕ полную рекурсивную
  `_collect_referenced_names`, она тоже осталась в файле, но НЕ используется
  для этой проверки). Полная рекурсивная версия ловит МАССУ ложных
  срабатываний на безобидном переиспользовании одного JVM-слота/имени в
  НЕЗАВИСИМОЙ (непересекающейся) более поздней ветке if/else - на этом
  дважды падала успешность на 8-14 п.п. НЕ ТРОГАТЬ эту логику без прогона
  полной регрессии (см. HANDOFF_1).

  ЕЩЁ ВАЖНЕЕ: при первой попытке чинить это огрёб ещё и банальный
  `NameError: ThrowStmt not defined` (забыл добавить импорт в начале
  engine.py). Эта ошибка ловилась общим `except Exception` в
  `decompile_method_body` и маскировалась под "внутренняя ошибка
  декомпилятора" в статистике - что и создало ЛОЖНОЕ впечатление "проверка
  ложно срабатывает" в первый раз (на самом деле она вообще не успевала
  выполниться - крашилась раньше). Мораль: при разборе непонятных
  статистических просадок после правки - СНАЧАЛА смотри
  `stats.fallback_reasons` в README (какая ИМЕННО причина отката выросла),
  не гадай и не откатывай фикс вслепую.
- `null`/`this` как псевдо-типы протекали в объявления переменных (`null
  __stk1;`, `this var7 = this;`) - было ЧЕТЫРЕ разных места с этим багом
  (engine.py stack-crossing, stackvm.py materialize/_refine_type,
  structure.py ternary-fold, engine.py refresh-crossing-types) - все
  исправлены через общий `_PSEUDO_TYPES = ("null", "this")` (stackvm.py).
- crossing-temp (`__stkN`) объявления не включались в static{} блок класса
  (были только в теле обычных методов) - исправлено.

## Дженерики/эрозия типов (без полного парсинга Signature)

- `_coerce_arg` (stackvm.py) - если тип значения на границе вызова - наш
  маркер эрозии `"Object"`/`"java.lang.Object"` (ВНИМАНИЕ: это ДВЕ РАЗНЫЕ
  строки в разных местах кода - `method_descriptor_to_java` возвращает
  fully-qualified `"java.lang.Object"`, а внутренние дефолты типа
  `default_type_for_cat` - голое `"Object"` - при сравнении ВСЕГДА проверяй
  ОБЕ формы, на этом уже спотыкались), а РЕАЛЬНЫЙ (из дескриптора вызываемого
  метода) тип параметра конкретнее - вставляется явный каст. Безопасно (раз
  байткод скомпилировался - рантайм-тип точно совпадает).
- То же самое применено к лямбда-синтезу (`_build_lambda` в stackvm.py) -
  и к аргументам синтетического метода, и к RECEIVER'у при instance-method-
  reference (`Map.Entry::getKey` и т.п.) - раньше ловился каскад
  "incompatible types: Object cannot be converted to X" / "cannot find
  symbol: method X() on Object".
- Оставшийся пробел: НЕ парсим Signature-атрибут вообще, поэтому объявления
  полей/переменных с generic-параметрами (`Map<String,UUID>`) всё ещё
  печатаются как raw `Map` - касты на границах вызовов это компенсируют
  через описанный выше механизм, но КРАСИВЕЕ было бы парсить Signature.
  Не делали - большая отдельная фича (см. HANDOFF_3).

## Синтетические конструкции

- Абстрактные/native методы (`m.code is None`) печатались с пустыми `{ }` -
  невалидно для интерфейсов/abstract классов. Теперь `;`.
- `lambda$xxx$N` синтетические методы резались по последнему `$` эвристикой
  обфускации (для классов `Foo$1` это правильно, для методов - НЕТ, давало
  `base="0"` длины 1 -> ложно "обфусцировано" -> переименовывалось в
  бессмысленный `methodN`). Теперь `$` в имени МЕТОДА/ПОЛЯ вообще не
  флагуется как обфускация (всегда синтетика компилятора).
- `interface X { static FIELD; static {...} }` - невалидно (JLS: initializers
  not allowed in interfaces) - для простых случаев (одно прямое
  присваивание на поле) инициализатор инлайнится прямо в объявление поля;
  для сложных - честный комментарий-предупреждение вместо гарантированной
  ошибки компиляции.
- `synchronized(x){}` не сворачивается вообще (SyncStmt определён, но нигде
  не конструируется - настоящая свёртка monitorenter/monitorexit НЕ
  реализована, это отдельная большая фича) - если найден несвёрнутый монитор,
  честно откатываемся на байткод (`_contains_unfolded_monitor`) вместо
  печати кода с полностью потерянной семантикой блокировки.
- `package-info.java` печатался как `interface package-info {}` (невалидно) -
  теперь только `package x.y.z;` (+ package-level аннотации, если есть).

## Аннотации (последняя сессия)

- `classfile.py` парсит `RuntimeVisibleAnnotations`/`RuntimeInvisibleAnnotations`
  (класс/поле/метод) и `RuntimeVisible/InvisibleParameterAnnotations`
  (параметры). `@NotNull`/`@Nullable`/любые другие аннотации теперь
  печатаются (main.py::`_format_annotation`) - marker-аннотации полностью,
  аннотации с простыми (строка/число/bool) аргументами - тоже, сложные
  (enum-константы/вложенные аннотации/массивы) - печатается только маркер
  без аргументов (не гадаем на сложных случаях).

  ВАЖНО, ЧТОБЫ НЕ НАСТУПИТЬ ПОВТОРНО: JetBrains `@NotNull`/`@Nullable` имеют
  `RetentionPolicy.CLASS` - лежат в **Invisible**, не Visible - парсер уже
  это учитывает (мержит оба атрибута), но если будешь трогать этот код и
  вдруг решишь "оптимизировать" и читать только Visible - сломаешь именно
  то, ради чего это всё писалось.

## Библиотеки/ресурсы

- Известные бандленные библиотеки (`pom_builder.KNOWN_LIBS`) НЕ
  декомпилируются - удаляются из class_files ДО сканирования
  external_dotted, попадают в pom.xml как maven-`<dependency>` через
  существующий механизм угадывания зависимостей. НЕ копируются как .class -
  доверяем maven'у их подтянуть.
- Их РЕСУРСЫ (нативные .so/.dll/.dylib, .properties, META-INF/maven/*)
  ТОЖЕ теперь пропускаются (раньше просачивались в resources несмотря на
  то что классы пропускались - находили 23 МБ мусора на одном jar).
- Релоцированные (maven-shade `<relocations>`) библиотеки ТОЖЕ
  распознаются - `pom_builder.parse_shade_relocations()` читает
  `<relocations>` из pom.xml плагина, `main.py::_relocated_library_prefixes`
  сопоставляет с KNOWN_LIBS.
- `find_pom_properties_and_xml` раньше слепо брала ПОСЛЕДНИЙ найденный
  `pom.xml`/`pom.properties` в jar'е - в шейднутом jar'е таких МНОГО (один
  на библиотеку) - могла подхватить pom.xml БИБЛИОТЕКИ вместо плагина.
  Теперь выбирает тот, чей artifactId НЕ совпадает ни с одной известной
  библиотекой.

## GUI (Windows-only, gui.py)

- CustomTkinter (тёмно-синяя тема, скруглённые виджеты) с graceful fallback
  на classic ttk-тему, если `customtkinter` не установлена (`pip install
  customtkinter`).
- Кнопка "Открыть проект в VS Code" (ищет `code`/`code.cmd` в PATH).
- Иконка приложения генерируется в памяти через PIL (никаких внешних
  файлов-картинок) - `{`/`}` рисуются ОТДЕЛЬНО (не одной строкой `"{ }"` -
  кернинг непредсказуем, был баг с перекошенной иконкой), банка по центру
  между ними, звезда в свободном углу.
- Карточка-сводка по jar (размер/Java-версия/классы/пакеты/имя плагина) -
  появляется после выбора файла.
- Toast-уведомление "Готово, N файлов" в углу вместо блокирующего диалога.
- Консоль тоже цветная (ANSI, только если реальный tty - см.
  `main.py::classify_line`/`cprint`/`_supports_color`), с живым
  прогресс-баром (`\r`-перерисовка) на этапе декомпиляции классов.
- `_quality_rating()` (verify.py) - "крутизна декомпиляции" по проценту
  успеха: >=96.9% "идеально", >=90% "отлично", >=75% "неплохо", иначе
  "так себе"; <50% - специальное сообщение "это баг, напиши в Telegram
  @ERROR_92" (пользователь попросил именно так).
